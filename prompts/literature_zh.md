# Graphiti：为 AI Agent 构建时序上下文图

## 一句话理解

开源的时序知识图谱引擎，给长期对话型 Agent 装"有时间维度"的记忆系统。

## 它解决什么问题

传统 RAG 把知识切片塞向量库，回答时拉相似片段，问题是知识是"静态快照"——昨天的事实和今天的事实没有先后概念，更新一条信息要全量重建索引，查询拿到的是孤立文档而非结构化关系。

对长期对话型 Agent（客服、个人助手）这是致命的：用户三个月前说的偏好今天还要不要遵守？同一个实体的属性反复变化怎么处理？纯向量库给不出答案。

Graphiti 的解法是不做向量倒排，而是维护一张**随时间演化的知识图谱**。每次输入作为"Episode"被吸收，自动抽实体和关系，事实带有效时间窗口（`valid_at` / `invalid_at`）。

## 核心机制（用自己的话）

- **Entity（实体）**：图谱节点，附带随时间演化的摘要
- **Episode**：原始事件，可以是对话片段、JSON、文档
- **Edge / Fact**：实体间关系，关键是带时效窗口
- **Temporal Reasoning**：能查询"2024 年 3 月时 X 和 Y 是什么关系"
- 数据流：Episode 进入 → LLM 抽 entity & relation → 与已有图谱合并 → 写入 Neo4j

## 同类对比与差异化

| 对照对象 | 差异 |
|---------|------|
| 纯向量库（Pinecone/Milvus） | 向量库丢失结构和时间维度，Graphiti 在两个维度上都补足 |
| Zep（同公司前作） | Zep 是会话级记忆，Graphiti 是知识级；Graphiti 更通用但部署重 |
| 通用知识图谱（Neo4j 裸用） | Neo4j 是底座，需要手写抽取流水线，Graphiti 是完整的"Episode 进 → 图谱出" |

## 落地场景与二开想象

1. **长期客服 Agent**：客户的偏好、历史问题、合同状态随时间变化，用 Graphiti 替代 RAG 能避免"客户三个月前说不要再发短信，现在又被发"的事故。二开重点：把 CRM 数据源接到 Episode 接口
2. **个人知识助手**：读完书/文章后形成"我对某个主题的演化观点"图谱，回答问题时能区分"我以前认为 vs 现在认为"。二开重点：跟 Obsidian/Notion 整合，做笔记自动入图
3. **企业内文档问答**：政策、流程、组织架构都会随时间变化，Graphiti 能回答"2023 年这个流程是什么、现在改成什么了"。二开重点：把 Confluence/Lark 文档变更事件 stream 进来

## 生产可用性评估

- **依赖**：Neo4j（部署成本高于纯向量库）+ LLM API（用 GPT-4 做实体抽取，token 成本不低）
- **社区**：基于 GitHub Stars 增长曲线，活跃度尚可，文档完整度中等
- **生产差距**：实体抽取的 LLM 成本是规模化的瓶颈；中文 NER 准确率需要自测验证；图谱压缩/裁剪策略文档薄弱
- **可投产场景**：内部 POC、中小型 Agent 应用；大规模生产需要先压测成本

## 我的判断

值得跟进，但**不是现在直接上**。先在内部记忆助手 POC 上验证，等 Neo4j 部署经验+成本预算明朗后再考虑给客户上。中文场景下要自己做 NER 优化或者换抽取链路。
